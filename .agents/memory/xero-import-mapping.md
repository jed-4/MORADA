# Xero bill import: cost-code mapping and attachment download

**Tracking category → cost code:** Xero connection has `trackingCategory1Id` (TC1 = cost codes) and `trackingCategory2Id` (TC2 = projects). Cost codes (shared/schema.ts) carry `xeroTrackingOptionId` / `xeroTrackingOptionName` / `xeroTrackingCategoryId`. On import, build a map `xeroTrackingOptionId → costCode.id` (and a name-based fallback map). Per Xero line item, scan `xl.Tracking[]`, prefer the entry whose `TrackingCategoryID === trackingCategory1Id` and match by `TrackingOptionID` (GUIDs — collision-free). The resolved cost code is carried ONLY via `costCodeId`; resolve order is matched-tracking → import-wide `defaultCostCodeId` → uncategorised. **`lineType` MUST stay `"custom"` for imported lines** — the `bill_line_type` pgEnum is `["estimate","item","custom"]`, so `"cost_code"` is NOT a valid value and crashes the insert (`invalid input value for enum bill_line_type`). `lineType` plays no part in budget attribution: `storage.recalculateBudgetLineItems` buckets bill actuals purely by `costCodeId`.
**Why the GUID path is safe regardless of TC1:** option IDs are globally unique, so the name fallback (its only collision risk) only matters when TC1 is unknown.

**Attachment download:** there was no download method — added `xeroService.downloadInvoiceAttachment(connectionId, xeroInvoiceId, att)`: GET `/Invoices/{id}/Attachments/{AttachmentID | encoded FileName}` with `Accept: <MimeType>`, returns a Buffer. List via existing `getInvoiceAttachments`. Persist by uploading the buffer with `new ObjectStorageService().uploadObjectEntity(buffer, contentType, companyId)` (returns `/objects/company/<id>/...`), then `storage.appendBillAttachment(billId, {objectPath, filename, mimeType, uploadedAt, uploadedBy, source:"xero"})`. `BillAttachment.source` enum allows `"xero"`. Do it best-effort per file so one failure doesn't abort the import.

**Import-preview job filter must page through ALL bills:** Xero cannot filter `/Invoices` by tracking category server-side — tracking lives on line items. So filtering by job/tracking on a single page (the default 100 most-recent ACCPAY, Date DESC) silently drops a job's older bills. **Why:** user-reported symptom "not all bills for the job show on re-import" was this, NOT the hard-delete (delete correctly clears `getBillByXeroId` so bills re-import fine). **How to apply:** when a tracking filter is active, load every page (loop `listBills` until a short page; cap maxPages) then filter locally; keep the fast single-page path for the unfiltered "all jobs" view. Any new server-side filter that lives on line items (not invoice headers) has the same constraint.

**Supplier resolution is per-row, not global:** the import modal shows each Xero bill's BuildPro supplier match (exact `xeroContactId` link, or high-confidence fuzzy via `matchSupplier`) and forces the user to LINK or CREATE for unmatched rows — the old global "Skip/Auto-create" toggle silently dropped bills whose supplier wasn't mapped (root-cause of "Kenj Clad & Roofing" vanishing). The client sends `supplierResolution: {xeroInvoiceId → {mode:"link"|"create", supplierId?}}`; in the import loop this resolution takes precedence over id-match → fuzzy → legacy fallback. Link/create backfills the contact's `xeroContactId` so future imports auto-match. **Gotcha:** `storage.getContacts(companyId)` returns ALL contact types — fuzzy candidate lists filter to `contactType === "supplier" || !contactType`, but the exact-`xeroContactId` match (preview + import) is intentionally left UNFILTERED so preview and import agree; a non-supplier contact sharing a Xero ContactID could therefore be matched as a bill supplier. **Batch dup guard:** create-mode re-checks the per-iteration `contacts` fetch for an existing `xeroContactId` before creating, so repeated bills for the same new supplier in one sequential import reuse one contact (concurrent requests can still race without a DB uniqueness constraint).

**Xero locks invoices that have payments — never push line/status edits to them:** once a Xero invoice has any payment (or credit note) allocated, Xero rejects line-item/status updates with "To update fields on a paid invoice line item, you must supply a LineItemID" and "The status AUTHORISED cannot be applied … it has payments". `xeroService.updateBill` sends line items WITHOUT LineItemID and forces `Status: AUTHORISED`, so it cannot update a paid/partially-paid invoice. **How to apply:** in `pushBillToXeroInternal`, before any tracking/preflight/updateBill work, if `bill.xeroInvoiceId && (bill.status === "paid" || bill.paidAmount > 0)` skip the push, clear any stale `xeroLastSyncStatus: "failed"`, and return `INVOICE_LOCKED` (client shows a gentle "Bill saved", not a red error). The local edit (e.g. a new cost code) still saves and feeds the budget — it just isn't mirrored to the locked invoice. Unpaid linked invoices are fine: Xero replaces line items without LineItemID. **Caveat:** gating on local `paidAmount` misses payments applied directly in Xero after import while BuildPro still shows 0 — harden later by fetching live `AmountPaid` only when local says unpaid.

**Attachments need their own OAuth scope:** listing/downloading/uploading Xero invoice attachments requires `accounting.attachments` in `XERO_SCOPES` (read+write; `.read` is read-only). Without it every attachments call returns `401 AuthorizationUnsuccessful` even though invoice reads work — this was the real cause of "imported bills have no files", NOT the download code. **How to apply:** changing `XERO_SCOPES` does NOT upgrade existing tokens — the user must disconnect+reconnect Xero to mint a token with the new scope, and already-imported bills won't backfill attachments (re-import or a dedicated backfill is needed).

**Xero rate limit (429) — every per-bill call path must back off, not just the page loop:** Xero enforces ~60 API calls/min/tenant. The import burst plus background pollers (xeroPoStatusSync etc.) on the same tenant easily exceeds it; one 429 cascades — it rate-limits the whole tenant so the *next* import-preview load also 429s and the modal hangs on "Loading Xero bills…". **Why the attachment feature was the regression:** the import loop runs PER BILL `getInvoice` + `getInvoiceAttachments` + `downloadInvoiceAttachment` (per file). Adding attachment download roughly TRIPLED per-bill calls, so a 20–30 bill import jumped from ~25 to 60–90 calls in one burst and started 429-ing where it never used to. **How to apply:** (1) route EVERY Xero call through module-level `xeroFetchWithRetry(url, init, {maxRetries, label})` (retries 429 honoring `Retry-After`, capped ~60s, bounded retries) — it is easy to miss one; `getInvoice` was left as a raw `fetch` and silently undercut the whole fix. (2) Pace the import loop (~1s/bill) so a batch stays under the limit proactively, with retry as the safety net. (3) Make preview calls FAIL-FAST not patient: `getTrackingCategories {maxRetries:0}` (non-fatal — dropdown just stays empty) and `listBills/listAllBills {maxRetries:1}`, so an already-rate-limited tenant errors the modal in seconds instead of hanging minutes. (4) Background pollers must batch: fetch all POs via one paged `listAllPurchaseOrders` into a Map, not one GET per linked PO. Do NOT fix 429 by lowering `maxPages` — that re-breaks the show-all-bills filter. **Known limitation:** synchronous import is still timeout-prone for very large batches (pace + possible 60s retry waits stack); the real fix is a background attachment-fetch job, deferred.

**Attachment import failures must surface, never silently drop:** attachment download is best-effort per file so the bill still imports, but a swallowed failure means "imported bills have no files" with no signal. Collect a per-bill `attachmentWarning` in the import results; a 401/403 on the attachments call almost always means the Xero connection predates the `accounting.attachments` scope → warn the user to reconnect Xero in Settings. The client surfaces the warning in the "Import complete" toast.
